## Read this first: what already existed vs. what this session added

Before writing any code, this session audited the repository and found that most of the
commercial pipeline described in the automation brief (sections A–H, K) **already exists**, is
already tested (327 pre-existing tests), and is already default-safe. The task actually facing
this session was smaller than "build the whole loop from scratch": it was to find the genuine
gaps, fill them without duplicating what already works, and wire everything into one coherent,
safety-gated automation layer.

## Two independent gate layers — do not conflate them

1. **Capability gates** (pre-existing, untouched): `outbound.enabled`/`dryRun`/`liveSendApproved`,
   `inbound.enabled`, `discovery.enabled`, `revenue.publicIntake`, etc. These govern whether a given
   side effect (a real send, a real Gmail read, a real discovery query) can *literally happen at
   all*. They default to the safest possible state and require explicit production-only
   configuration to relax (see `validateStartupConfig` in `src/config.mjs`).
2. **Automation gate** (new, `src/automation/mode.mjs`): governs whether the scheduler is allowed
   to *advance a prospect through a stage without a per-item owner click*. `AUTOMATION_MODE=shadow`
   (the default) never advances anything automatically — every stage still runs and records its
   result, but nothing is sent, charged, or acted on without the owner. `approval` mode requires an
   active signed campaign policy before automated advancement is allowed within that policy's
   caps; the owner still only sees exceptions and the daily digest. `autonomous` mode additionally
   requires `AUTOMATION_AUTONOMOUS_CONFIRMED=true` and is refused in the test environment.

A real send, charge, or site change requires **both** layers to independently allow it. Neither
implies the other, and this session did not weaken either.

## Why fulfillment (site changes) stays manual by design

Per `src/delivery.mjs` (pre-existing) and `src/automation/fulfillment.mjs` (new), no code path in
this repository can modify a customer's website automatically. `automaticModificationAllowed` is
hardcoded `false` in every delivery record and is never set from any input. Implementation work
requires `written-authorization` and `access-provisioned` customer inputs to reach `received`
status (an owner action) before the `implement-scope` checklist item can even move to
`in-progress`. This is intentional and is not something the "AUTONOMOUS" mode is meant to remove.

## What SHADOW / APPROVAL / AUTONOMOUS actually do in this codebase

- **SHADOW** (default): `automationStatus(cfg).live === false`. Every stage worker still runs
  (crawl, score, qualify, draft) and records its result, because none of that requires the
  automation gate — it always ran this way even before this session, gated only by the
  capability gates above. What's new is that `isShadowOnly(cfg)` is now the single predicate a
  caller checks before doing anything an outside party would see.
- **APPROVAL**: `evaluateAutomationGate` requires `hasActivePolicy: true` — i.e. a
  `isCampaignPolicyActive()` policy for that campaign. Inside an active policy's caps
  (`dailySendCap`, `hourlySendCap`, evidence/confidence thresholds, allowed hours), the existing
  campaign engine (`campaign.autoSend`) may already process prospects without a per-item click;
  this session's contribution is making that a *policy*, not just a config flag: signed, versioned,
  expiring, with postal/payment-rail confirmation and explicit pause thresholds.
- **AUTONOMOUS**: same as APPROVAL plus `AUTOMATION_AUTONOMOUS_CONFIRMED=true`. This repository
  does not add any new capability under autonomous mode — no negotiation, no fulfillment
  automation beyond lane assignment, no new charge path. It only removes the mode-level block; the
  capability gates (outbound.enabled, etc.) are unchanged and still default to off.

## New collections

Two new collections were added additively to `src/store.mjs` and `migrations/012_automation_layer.sql`:
`fulfillmentTasks` (lane assignment + SLA + checklists) and `automationDigests` (persisted daily/
weekly digest history). Both back onto the same generic JSON/Postgres dual-backend store every
existing collection uses; no existing collection's schema or behavior changed.

## Not built in this session (see readiness doc for the full list)

- A real outbound provider beyond the existing Gmail integration (explicitly out of scope per the
  task).
- A real Apify network fetch is *implemented* (`defaultApifyFetcher`) but never called by anything
  in this repository — only the injectable-fetcher path is exercised by tests, and polling is
  refused unless `APIFY_ENABLED=true` with real credentials, which nothing in this session sets.
- An HTTP/UI surface for the owner to sign a campaign policy, enroll a monitoring subscription, or
  drive a fulfillment task through its checklist from a browser — the logic and validation exist
  and are tested; wiring them into `server.mjs`'s existing admin routes is unstarted.
- A literal `EVIDENCE_VERIFIED` status distinct from `CRAWLED`/`QUALIFIED` in the existing
  `acquisitionStatus` vocabulary — the underlying crawl+audit pipeline validates evidence in one
  pass (`src/qualification.mjs#validateAuditEvidence`) rather than exposing an intermediate status,
  so `projectLifecycleState` cannot currently distinguish "crawled" from "evidence verified" for
  already-running prospects. The formal state machine still defines and tests the state and
  transition.
